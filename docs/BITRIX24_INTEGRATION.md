# Интеграция с Bitrix24 — исследование и план

> Заметки по возможной интеграции нашего приложения управления задачами (MBC Task Management) с Bitrix24.
> Автор: Alisher. Дата: 22.05.2026.
> Статус: черновик/исследование. Решение по реализации пока не принято.

## 1. Краткий вывод

- Интегрироваться можно через **вебхуки Bitrix24** — это проще, чем полноценное OAuth-приложение, и подходит для одного аккаунта.
- **Протестировать можно бесплатно**: REST API формально доступен на платных тарифах, но на бесплатном облачном портале можно включить **демо-режим** и тестировать.
- **Наши статусы задач совпадают со статусами Bitrix24 почти 1:1** — это сильно упрощает синхронизацию.
- Наш стек (Node/Express + PostgreSQL + уже работающий приём вебхуков от Telegram) идеально подходит для приёма событий от Bitrix.

## 2. Бесплатный доступ для теста

Официально: «The REST API in Bitrix24 is available only on paid plans», НО на бесплатном аккаунте можно активировать **демо/триал** и тестировать API до покупки.

Варианты:
- **Облако (проще всего):** регистрируем бесплатный портал `нашакомпания.bitrix24.ru`, активируем демо → REST и вебхуки открываются.
- **Коробка (On-Premise):** есть бесплатная триал-версия для локального развёртывания — тяжелее, для теста интеграции не нужно.

⚠️ Уточнить срок демо (обычно ~15–30 дней).

## 3. Откуда брать API-токен (входящий вебхук)

Токен — это секрет внутри URL вебхука. Получается прямо в интерфейсе портала.

1. Зарегистрировать портал на bitrix24.ru / bitrix24.com.
2. Активировать демо: верхнее меню → виджет «Мой план» → **«Активировать демо»**.
3. Левое меню → **Приложения → Разработчикам** (*Applications → Developer resources*).
4. Вкладка **«Другое» → «Входящий вебхук»** (*Incoming webhook*).
5. Отметить права (scope): минимум **`task`** (+ `user`, `sonet_group` для маппинга сотрудников и проектов).
6. Система выдаст готовый URL:
   ```
   https://нашпортал.bitrix24.ru/rest/1/<секретный_токен>/
   ```
   Этот URL целиком — ключ. К нему дописываем метод: `.../tasks.task.add.json`.
   OAuth не нужен, ключ **бессрочный**.

### Проверка ключа за 1 минуту
Открыть в браузере:
```
https://нашпортал.bitrix24.ru/rest/1/<токен>/profile.json
```
Если вернётся JSON с профилем — ключ рабочий.

## 4. События из Bitrix → к нам (исходящий вебхук)

Там же, вкладка **«Исходящий вебхук»**:
- указываем наш handler-URL (например `/webhooks/bitrix`);
- выбираем события: `OnTaskAdd`, `OnTaskUpdate`, `OnTaskDelete`;
- Bitrix выдаёт **токен приложения** — им проверяем подлинность входящих POST-запросов.

Формат: при событии Bitrix шлёт POST с данными события, изменёнными полями, временной меткой и токеном.

---

## 5. Структура запроса (входящий вебхук)

```
https://[домен]/rest/[user-id]/[секрет]/[метод].json?[параметры]
```
Пример: `.../rest/1/abc123/tasks.task.add.json`

- `/rest` — указывает на REST-вебхук
- `/1` — ID пользователя, создавшего вебхук
- `/abc123` — секрет (конфиденциальный)
- `tasks.task.add` — метод
- `.json` — формат (по умолчанию JSON)

---

## 6. Основные методы задач (REST API)

| Метод | Назначение |
|---|---|
| `tasks.task.add` | Создать задачу |
| `tasks.task.update` | Изменить задачу |
| `tasks.task.get` | Получить одну задачу |
| `tasks.task.list` | Список задач |
| `tasks.task.delete` | Удалить задачу |
| `tasks.task.complete` / `start` / `pause` | Смена статуса |
| `tasks.task.delegate` | Переназначить (сменить ответственного) |
| `tasks.task.history.list` | История изменений |
| `task.checklistitem.*` | Чек-листы |
| `task.commentitem.*` | Комментарии (legacy; в новых версиях — чат задачи) |
| `tasks.task.files.attach` | Прикрепить файлы из Диска |

**События:** `OnTaskAdd`, `OnTaskUpdate`, `OnTaskDelete`.
**Scope:** требуется `task`.

### Ключевые поля задачи Bitrix
- `TITLE` — название
- `DESCRIPTION` — описание
- `RESPONSIBLE_ID` — ответственный (ID пользователя)
- `DEADLINE` — срок
- `STATUS` — статус
- `GROUP_ID` — проект/группа
- `PRIORITY` — приоритет (0 низкий / 1 обычный / 2 высокий)
- `ACCOMPLICES` — соисполнители
- `AUDITORS` — наблюдатели

## 7. Сопоставление статусов (совпадают почти 1:1)

| Наш статус | Bitrix `STATUS` |
|---|---|
| Новый | `2` (Ждёт выполнения) |
| В процессе | `3` (Выполняется) |
| Проверка | `4` (Ждёт контроля) |
| Закрыт | `5` (Завершена) |

`6` (Отложена) — у нас не используется.

## 9. Идеи интеграции (по возрастанию сложности)

1. **MVP — односторонняя выгрузка (мы → Bitrix24).** При создании/изменении задачи у нас вызываем `tasks.task.add`/`update`. Храним `bitrix_task_id`. Низкий риск, быстро.
2. **Синхронизация статусов и сроков.** Минимальный двусторонний обмен только по `STATUS`/`DEADLINE` — самое ценное и наименее конфликтное.
3. **Полная двусторонняя синхронизация.** Приём исходящих вебхуков Bitrix (`OnTaskUpdate`) на `/webhooks/bitrix` → обновление Postgres → пуш через наш realtime (`src/sync/realtime.js`). Нужна **защита от петель** (метка источника изменения) и разрешение конфликтов по времени (у нас уже есть `snapshot-merge`).
4. **Импорт из Bitrix.** `tasks.task.list` + `user.get` + `sonet_group.get` для подтягивания пользователей и проектов и построения таблиц соответствия.
5. **Комментарии и история.** Причину переназначения/комментарии → `task.commentitem.add`; тянуть `tasks.task.history.list`.
6. **Глубокие ссылки.** На карточке задачи кнопка «Открыть в Bitrix24» (просто ссылка по `bitrix_task_id`). Самое дешёвое и сразу полезное.

**Рекомендация:** начать со связки **1 + 6** — выгрузка + кнопка-ссылка. Видимый результат за минимум кода, закладывает `bitrix_task_id`, на который потом нарастёт двусторонняя синхронизация (идея 3).

## 10. Что учесть при реализации

- **Лимит запросов** ~2 req/сек. Для массовых операций — метод **`batch`** (до 50 команд за вызов).
- **Таблицы соответствия:** сотрудник (по телефону) → Bitrix `user_id`; объект → проект/группа. Без них не выставить ответственного.
- **Секреты** — в env-переменные на Railway (как `JWT_SECRET`):
  ```
  BITRIX_WEBHOOK_URL=https://нашпортал.bitrix24.ru/rest/1/<токен>/
  BITRIX_OUTGOING_TOKEN=<токен_исходящего_вебхука>
  ```
- **Не выкладывать секретный URL в публичный git** — он даёт полный доступ под пользователем.
- **Версии API:** есть REST и REST 3.0. Для задач берём документированный REST (`tasks.task.*`).
- **Где жить коду:** новый модуль `server/bitrix.js` (клиент к API) + эндпоинт приёма событий по аналогии с `server/telegramWebhook.js` + миграция БД под `bitrix_task_id`.
